# v2.4.2 ŌĆö Yahboom ROS 2 OmniGraph Topic Publishers

## Purpose

v2.4.2 adds the OmniGraph action graph to the Yahboom visible placeholder stage.
When complete, `verify_yahboom_live_topics.py --strict` will pass and Gate A5
will be open for the first valid rosbag2 recording.

No rosbag is recorded in v2.4.2. No training data is claimed. This milestone
proves that all five canonical ROS 2 topics are live from Isaac Sim.

## What v2.4.1 Left Incomplete

After loading `yahboom_m3pro_visible_placeholder.usda` and pressing Play in
v2.4.1, `ros2 topic list` shows only:

```
/parameter_events
/rosout
```

The ROS 2 Bridge extension is enabled, but no OmniGraph action graph exists to
wire the simulation prims to ROS 2 publishers. The bridge is on but nothing is
connected to it.

## The Five Required OmniGraph Nodes

One action graph containing these six nodes (one tick source, five
publisher/subscriber nodes) must be added:

| Node name | Type | Wiring | Topic |
|---|---|---|---|
| `OnPlaybackTick` | `omni.graph.action.OnPlaybackTick` | Drives all publishers | ŌĆö |
| `ROS2Context` | `omni.isaac.ros2_bridge.ROS2Context` | Tick ŌåÆ ROS 2 runtime | ŌĆö |
| `ROS2PublishImage` | `omni.isaac.ros2_bridge.ROS2PublishImage` | `camera_link/sensor` prim | `/camera/image_raw` |
| `ROS2PublishOdometry` | `omni.isaac.ros2_bridge.ROS2PublishOdometry` | `base_link` prim | `/odom` |
| `ROS2PublishTransformTree` | `omni.isaac.ros2_bridge.ROS2PublishTransformTree` | `YahboomM3Pro` prim tree | `/tf` |
| `ROS2PublishLaserScan` | `omni.isaac.ros2_bridge.ROS2PublishLaserScan` | `lidar_link` prim | `/scan` |
| `ROS2SubscribeTwist` | `omni.isaac.ros2_bridge.ROS2SubscribeTwist` | `/cmd_vel` | wheel joints |

The `ROS2Context` node initialises the ROS 2 runtime inside Isaac Sim. Without
it, publishers may silently fail depending on the Isaac Sim version.

## Method A ŌĆö Python Script (Recommended)

Run the OmniGraph creation script from Isaac Sim's built-in Python interpreter.

### Step 1 ŌĆö Open Isaac Sim and load the stage

```bash
~/.local/share/ov/pkg/isaac-sim-*/isaac-sim.sh
```

File ŌåÆ Open ŌåÆ `assets/robots/yahboom_m3_pro/yahboom_m3pro_visible_placeholder.usda`

Confirm ROS 2 Bridge extension is enabled (Window ŌåÆ Extensions ŌåÆ search
"ROS2 Bridge").

### Step 2 ŌĆö Run the OmniGraph script

Open the Script Editor in Isaac Sim (Window ŌåÆ Script Editor), then run:

```bash
~/.local/share/ov/pkg/isaac-sim-*/python.sh \
  scripts/gnm/add_yahboom_ros2_omnigraph.py
```

Or paste the script contents directly into the Script Editor and click Run.

The script creates the action graph, connects all nodes, sets all topic names,
and prints the graph path on success.

### Step 3 ŌĆö Press Play and verify

1. Press Play in the Isaac Sim toolbar.
2. In a separate terminal:

```bash
python3 scripts/gnm/verify_yahboom_live_topics.py --strict
```

Expected output after successful OmniGraph setup:
```
[OK]      /camera/image_raw
[OK]      /odom
[OK]      /tf
[OK]      /scan
[OK]      /cmd_vel
[OK] All canonical topics present. Recording gate is open.
```

### Step 4 ŌĆö Save the stage

File ŌåÆ Save (overwrite `yahboom_m3pro_visible_placeholder.usda`).

This persists the OmniGraph nodes inside the USDA so they reload automatically
next time.

## Method B ŌĆö GUI (Manual)

Use Isaac Sim's Visual Scripting interface. This is slower but works when the
Python script cannot be run.

1. Window ŌåÆ Visual Scripting ŌåÆ Action Graph ŌåÆ New Action Graph
2. Name the graph `/World/ROS2ActionGraph`
3. Add each node from the table above using the search bar
4. Set `topic_name` on each publisher node
5. Connect `OnPlaybackTick.outputs:tick` ŌåÆ `ROS2Context.inputs:execIn`
6. Connect `ROS2Context.outputs:execOut` ŌåÆ each publisher/subscriber `inputs:execIn`
7. Set prim paths on each node (see script for exact values)
8. Press Play and verify with `verify_yahboom_live_topics.py --strict`
9. File ŌåÆ Save

## Known Limitations of the Placeholder Stage

The visible placeholder stage (from v2.4.1) uses cube/cylinder primitives. This
affects what the publishers can actually send:

| Topic | Limitation |
|---|---|
| `/camera/image_raw` | Publishes from a synthetic camera render ŌĆö no photorealism, no real geometry |
| `/odom` | Odometry requires an articulation drive. Placeholder chassis has no drive joint. Values may be zero or static until physics is added. |
| `/tf` | TF publishes correctly from the USD prim tree ŌĆö `base_link`, `camera_link`, `lidar_link` will appear |
| `/scan` | LiDAR scan requires an Isaac Sim lidar sensor prim. Placeholder `lidar_link` is a cylinder. Scan may be empty until a real sensor is added. |
| `/cmd_vel` | Subscriber receives commands but cannot move the robot. No articulation drive exists in the placeholder. |

These limitations are acceptable for Gate A5. The goal is that all five topics
**publish** (even if the data is synthetic or static). `validate_yahboom_episode.py`
checks `message_count > 0` ŌĆö not data quality. Data quality matters for v2.5+.
