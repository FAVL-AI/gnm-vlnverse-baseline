# Real-Run Safety Certification

This document explains the three certificate tiers produced by FleetSafe
for real robot runs, their relative strength, and how to generate them.

---

## Certificate tiers

| Tier | How generated | qp_status | Strength |
|------|--------------|-----------|---------|
| **Runtime certificate** | CBF-QP filter running live | `optimal` / `estop_fallback` | ★★★ Full formal guarantee |
| **Posthoc bag certificate** | Reconstructed from recorded bag | `posthoc_observation` | ★★ Observational verification |
| **Telemetry-only certificate** | Bridge without CBF-QP | `not_available` | ★ Operational monitoring only |

---

## Tier 1 — Runtime certificate (strongest)

Generated by the safety filter *at the moment the command is computed*.

**What it proves:**
- The CBF-QP was solved (or the emergency stop fired) before the command was sent.
- Every command `u_safe` satisfies the CBF constraint `ḣ_i + α·h_i ≥ 0` within
  the numerical tolerance of the QP solver.
- The robot was inside the safe set `C = {x | h_i(x) ≥ 0 ∀i}` at time of issue.

**How to generate:**
Run the real-robot bridge *with* the CBF-QP filter and `--cert-log`:

```bash
python3 scripts/real_robot/run_m3pro_ros2_bridge.py \
    --cert-log results/certificates/runtime_real.jsonl
```

**What it does NOT prove:**
- Forward invariance between timesteps (gap between QP solves).
- Sensor accuracy (LiDAR range noise, blind spots).
- Actuator faithfulness (command sent ≠ motion executed).

---

## Tier 2 — Posthoc bag certificate

Reconstructed *after* the run from recorded scan data.

**What it proves:**
- The robot was observed at distance ≥ `d_safe` from all detected obstacles
  at every timestep where a `/cmd_vel` message was recorded.
- `safe = (min_dist_m >= d_safe)` is a *factual observation*, not a prediction.

**What it does NOT prove:**
- The original QP solve — there is no record of the QP being run.
- Safety between recorded scan updates.
- Anything about how the command was computed.

**How to generate:**
```bash
make certify-latest-real-bag    # finds latest m3pro_full_motion_* bag
# or manually:
python3 scripts/evaluation/certify_rosbag_run.py \
    --bag data/real_robot_bags/m3pro_full_motion_TIMESTAMP \
    --output results/certificates/posthoc_run.jsonl \
    --d-safe 0.5
```

**How to verify:**
```bash
make verify-latest-real-cert
# or:
python3 scripts/evaluation/verify_cbf_certificates.py \
    --input results/certificates/latest_real_robot.jsonl \
    --d-safe 0.5 --latency-ms 9999
```

The `--latency-ms 9999` flag disables the latency check — posthoc
certificates have `latency_ms=0.0` (the QP solve time is not recorded).

---

## Tier 3 — Telemetry-only certificate

Generated by the ROS2 bridge when it is used *without* a CBF-QP filter,
purely for distance monitoring.

**What it records:**
- Minimum LiDAR distance per `/scan0`/`/scan1` callback.
- Whether `min_dist_m >= d_safe` (flagged as `safe`).
- `qp_status = "not_available"` — explicitly marks the absence of formal filtering.

**How to generate:**
```bash
python3 scripts/real_robot/run_m3pro_ros2_bridge.py \
    --cert-log results/certificates/telemetry_only.jsonl
# (without a CBF-QP integration)
```

**Use case:** Operational safety monitoring, not formal verification.
A reviewer reading these certificates can see the robot was never closer
than `d_safe` but cannot attribute this to the safety filter.

## Full real-robot certification pipeline

```
record bag  →  certify_rosbag_run.py  →  verify_cbf_certificates.py  →  report
```

```bash
# One-shot pipeline:
make formal-real-report

# Step by step:
make certify-latest-real-bag   # → results/certificates/latest_real_robot.jsonl
make verify-latest-real-cert   # → pass/fail exit code
# (report is generated automatically by formal-real-report)
```

## Reading the certificate JSONL

Each line is a JSON object:

```json
{
  "timestamp": 1748149620.3,
  "model_name": "posthoc_bag",
  "u_nom": [0.15, 0.0],
  "u_safe": [0.15, 0.0],
  "h_min": 0.4375,
  "min_dist_m": 1.21,
  "cbf_active": false,
  "qp_status": "posthoc_observation",
  "constraint_margin_min": 0.4375,
  "latency_ms": 0.0,
  "safe": true,
  "notes": "posthoc certificate from recorded bag; verifies observed safety distance, not original QP solve"
}
```

Key fields:

| Field | Meaning |
|-------|---------|
| `h_min` | min barrier value: `min_dist_m² - d_safe²` (≥0 means safe) |
| `min_dist_m` | minimum LiDAR range reading (metres) |
| `cbf_active` | true if CBF-QP overrode the nominal command |
| `qp_status` | `optimal` / `estop_fallback` / `posthoc_observation` / `not_available` |
| `safe` | summary verdict: true if `min_dist_m >= d_safe` and QP OK |

## What to report to reviewers

When submitting results involving real-robot runs, report:

1. **Certificate tier** (runtime / posthoc / telemetry-only).
2. **Number of timesteps** certified.
3. **Number of violations** (`safe=false` rows).
4. **Minimum observed distance** across all timesteps.
5. **d_safe threshold** used (default 0.5 m).

A posthoc report from `make formal-real-report` contains all of the above
in a structured Markdown format suitable for appendix inclusion.
