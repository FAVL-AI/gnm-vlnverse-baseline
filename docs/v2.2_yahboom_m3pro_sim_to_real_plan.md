# v2.2 — Yahboom M3 Pro Sim-to-Real Topic Bridge Plan

## Why This Document Exists

v2.1 proved that Isaac Sim 5.1 can launch, enable its ROS 2 bridge, and publish
live ROS 2 topics on this machine. For that test we used **Nova Carter**, a large
research robot asset from NVIDIA, as the target.

Nova Carter was only a **smoke test** — a quick check that the plumbing works.

A smoke test asks: "Does the system power on at all?" It does not care which robot
is used. Nova Carter answered that question. The answer is yes.

From here on, Nova Carter is retired from this pipeline. The real target is:

```
Yahboom ROSMASTER M3 Pro
```

That is the physical robot sitting in the lab. Everything we build in simulation
should match what that robot can actually do. Training on the wrong robot means
the model will not transfer when we put it on the real hardware.

## Glossary — Every Term Explained Simply

### ROS 2 — Robot Operating System 2

**What it is:** Not an operating system like Windows. ROS 2 is a communication
framework — a shared intercom system that lets different programs on a robot
send and receive messages.

**What it does:** A robot runs many programs at once: one reads the camera, one
reads the lidar, one runs the navigation model, one drives the wheels. ROS 2 lets
those programs share data without needing to know anything about each other. They
post messages to named channels called **topics**.

**Standard or unique?** ROS 2 is the industry-wide standard for robotics research
and is used on thousands of different robots. Nothing about ROS 2 is specific to
our setup.

**Why it matters here:** GNM reads camera data from a ROS 2 topic. FleetSafe
reads lidar data from another ROS 2 topic. The robot receives movement commands
on a ROS 2 topic. ROS 2 is the backbone of the entire live pipeline.

---

### URDF — Unified Robot Description Format

**What it is:** A text file (written in XML, which looks like HTML with angle
brackets) that describes what a robot is made of.

**What it does:** A URDF lists every part of the robot — body, wheels, camera
mount, lidar mount — and says how those parts connect to each other. It gives
each part a name, a mass, a shape, and a position.

**Simple analogy:** If the robot were a piece of flat-pack furniture, the URDF
is the assembly diagram that shows which piece goes where and how they attach.

**Standard or unique?** URDF is the universal robot description format used
across all of ROS, ROS 2, Gazebo, Isaac Sim, and MuJoCo. It is a standard,
not something specific to this project.

**Why it matters here:** Isaac Sim needs a URDF to know what the Yahboom looks
like before it can simulate it. The URDF also defines the joint names
(`fl_wheel_joint`, `fr_wheel_joint`, `rl_wheel_joint`, `rr_wheel_joint`) that
every other part of the pipeline — the observation adapter, the safety filter,
the ROS 2 controller — references by name.

**Our URDF:** `assets/robots/yahboom_m3_pro/yahboom_m3pro.urdf`

---

### Xacro — XML Macro

**What it is:** A smarter version of URDF. Xacro lets you write the robot
description once with variables and macros, then generate the final URDF by
expanding those variables.

**Simple analogy:** If URDF is the final assembly diagram, Xacro is the
parametric template that lets you change the wheel radius in one place and have
it update everywhere automatically.

**Standard or unique?** Xacro is the standard way to write robot descriptions
for robots that share components or need to support multiple configurations.

**Our Xacro:** `ros2_ws/src/fleet_safe_description/urdf/yahboom_m3pro.urdf.xacro`
— includes a `use_sim` flag to switch between Gazebo simulation and real hardware.

---

### USD — Universal Scene Description

**What it is:** A 3D scene file format created by Pixar (the film studio) and
later adopted by NVIDIA for robotics simulation.

**What it does:** USD describes a 3D world — where the robot is, what the room
looks like, where the lights are, how materials look. Isaac Sim works natively
with USD files.

**Simple analogy:** If URDF is the robot's assembly diagram, USD is the full
blueprint of the factory floor where the robot operates.

**Standard or unique?** USD is an industry-standard format used in film
production and robotics. NVIDIA's adoption of it for Isaac Sim has made it the
dominant format for high-fidelity robot simulation.

**The URDF-to-USD step:** Isaac Sim cannot load a URDF directly into a live
scene in the same way it works with USD. The standard workflow is:

```
Yahboom URDF  →  Isaac Sim URDF Importer  →  Yahboom USD  →  Isaac Sim stage
```

**Our USD assets:**
- `assets/robots/yahboom_m3_pro/yahboom_m3pro_reference.usda` — reference stage
- `assets/robots/yahboom_m3_pro/yahboom_m3pro.usd` — imported USD
- Import instructions: `docs/YAHBOOM_URDF_TO_USD_IMPORT.md`

---

### rosbag2 — ROS 2 Recording

**What it is:** A tool that records every message published on every ROS 2 topic
to a file on disk, with precise timestamps.

**What it does:** If the robot runs for 30 seconds, the rosbag captures every
camera frame, every lidar scan, every odometry reading, and every movement
command that happened during those 30 seconds. You can replay the recording
later and it will re-publish all the topics exactly as they were.

**Simple analogy:** A rosbag is a flight data recorder (black box) for a robot.

**Standard or unique?** `rosbag2` is the standard recording tool built into
ROS 2. The format is not specific to this project.

**Why it matters here:** GNM training requires data. A rosbag episode recorded
from the Yahboom in Isaac Sim is how we produce that data. The episode is then
converted into GNM training format (context images + goal image + waypoint
labels) by `scripts/gnm/convert_rosbag_to_gnm_dataset.py`.

---

### TF — Transform Tree

**What it is:** A continuously updated description of where every part of the
robot is relative to every other part, and where the robot is in the world.

**What it does:** The camera sits at a specific position on the Yahboom body. The
lidar sits at a different position. TF publishes a constantly updated tree of
3D positions and rotations so that any program can answer: "where is the camera
right now, in world coordinates?"

**Simple analogy:** Imagine every sensor and every body part of the robot is
wearing a GPS tracker. TF is the map that shows where all those trackers are
relative to each other at every moment.

**Standard or unique?** `/tf` is a universal ROS 2 topic. Every robot that has
more than one sensor uses it.

**Why it matters here:** The GNM dataset converter uses TF to compute the robot's
exact pose at each frame, which becomes the ground-truth position for computing
waypoint labels. Without accurate TF, the (dx, dy) waypoint labels are wrong and
GNM training degrades.

**Yahboom TF frames:**
- `base_footprint` — the robot's footprint on the ground
- `base_link` — the robot's body centre (at wheel axle height)
- `camera_link` — the camera's position
- `lidar_link` — the lidar's position

---

### cmd_vel — Command Velocity

**What it is:** A ROS 2 topic where movement commands are sent.

**What it does:** A message on `/cmd_vel` contains two numbers:
- `linear.x` — how fast to move forward or backward (metres per second)
- `angular.z` — how fast to rotate left or right (radians per second)

The robot's motor controller reads this topic and drives the wheels accordingly.

**Standard or unique?** `/cmd_vel` with a `geometry_msgs/Twist` message is the
universal ROS 2 interface for any wheeled robot. It is not Yahboom-specific.

**Why it matters here:** `/cmd_vel` is the last step in the pipeline. GNM
computes what it wants to do. FleetSafe checks whether that is safe. FleetSafe
publishes the approved (or corrected) command to `/cmd_vel`. The robot moves.
Nothing drives the robot except via `/cmd_vel`.

---

### Mecanum Wheels — Why They Matter

**What they are:** Mecanum wheels are a special type of wheel with small rollers
mounted at 45-degree angles around the rim. Four of them together let the robot
move in any direction without turning first.

**What they do:** A normal two-wheeled robot must point its nose in the direction
it wants to go. A mecanum robot can slide left, slide right, or move diagonally
while facing forward. This is called **holonomic motion**.

**Why this matters for our robot:**
The Yahboom M3 Pro has four mecanum wheels:
- `fl_wheel_joint` — front left
- `fr_wheel_joint` — front right
- `rl_wheel_joint` — rear left
- `rr_wheel_joint` — rear right

The inverse kinematics (the maths that converts "I want to go forward at 0.2 m/s
and turn at 0.1 rad/s" into individual wheel speeds) for mecanum wheels is:

```
fl = (vx − vy − (lx + ly) × wz) / r
fr = (vx + vy + (lx + ly) × wz) / r
rl = (vx + vy − (lx + ly) × wz) / r
rr = (vx − vy + (lx + ly) × wz) / r
```

Where `lx` = half the wheelbase (0.0775 m), `ly` = half the track width
(0.0850 m), `r` = wheel radius (0.048 m).

**Why it matters for sim-to-real:** If we simulate a different-shaped robot (like
Nova Carter or TurtleBot, which use standard wheels), the physics of how it
moves will be wrong. GNM will learn from data that does not match what the real
Yahboom can do. When we transfer the trained model to the physical robot, it
will behave unexpectedly. Using the correct Yahboom model in simulation shrinks
this sim-to-real gap.

---

### Sim-to-Real — What It Means

**What it is:** The process of training or testing a robot in simulation, then
transferring what it learned to a real physical robot.

**Why there is a gap:** Simulation is always an approximation. Surfaces in
simulation are perfectly flat. Lighting is perfectly consistent. Physics is
simplified. The real world has floor irregularities, varying lighting, rubber
that grips differently depending on temperature, and motors that are slightly
different from each other.

**How we reduce the gap:**
1. Use the correct robot model (Yahboom, not Nova Carter).
2. Use the correct sensor positions (from the URDF).
3. Use realistic physics parameters (measured inertials, measured wheel radius).
4. Use the same topic names in simulation as on the physical robot.
5. Apply domain randomisation (varying physics parameters slightly during
   training so the model is robust to real-world variation).

## The Five Required Live Topics

These are the canonical topics that must be active for the full GNM + FleetSafe
pipeline to run on the Yahboom, whether simulated or real.

| Topic | Message type | Published by | Standard? | GNM uses it? | FleetSafe uses it? |
|---|---|---|---|---|---|
| `/camera/image_raw` | `sensor_msgs/Image` | Camera sensor | Yes — universal | Yes — primary input | No |
| `/odom` | `nav_msgs/Odometry` | Wheel encoders | Yes — universal | Yes — waypoint labels | Yes — current speed |
| `/tf` | `tf2_msgs/TFMessage` | TF publisher | Yes — universal | Yes — pose labels | Yes — coordinate alignment |
| `/scan` | `sensor_msgs/LaserScan` | Lidar sensor | Yes — universal | No | Yes — obstacle detection |
| `/cmd_vel` | `geometry_msgs/Twist` | FleetSafe (or manual) | Yes — universal | No (output, not input) | Yes — output |

### The Camera Topic Aliasing Issue

The Yahboom's real hardware ROS 2 driver publishes the camera on:
```
/camera/color/image_raw
```

The GNM pipeline and v2.1 verification use:
```
/camera/image_raw
```

These are different topic names. When moving from Isaac Sim to the real robot,
a topic remapping is required — either in the launch file or as a ROS 2 relay
node. The topic contract script (`check_yahboom_topic_contract.py`) documents
both the canonical name and the known alias.

This remapping must be in place before any real-robot rosbag recording can
produce valid GNM training data.

## How Yahboom Data Becomes GNM Training Data

```
Isaac Sim (Yahboom M3 Pro stage loaded)
         │
         │  Camera, lidar, TF, odometry all publishing
         │
         ▼
     rosbag2 records episode
         │
         ▼
convert_rosbag_to_gnm_dataset.py
         │
         ├── context_images/  (PNG frames from /camera/image_raw, resized)
         ├── goal_image.png   (frame from goal position)
         ├── waypoints.npy    (dx, dy displacements from /odom + /tf)
         ├── scan_summary.json (obstacle proximity from /scan)
         └── success_label.json (did the episode reach the goal?)
         │
         ▼
GNM training
         │
         ▼
Trained model + temporal stop head
         │
         ▼
FleetSafe safety evaluation
         │
         ▼
Real Yahboom robot deployment
```

## How FleetSafe Sits Before /cmd_vel

At the moment (v2.1), `/cmd_vel` is driven manually or by test publishers.
Once GNM and FleetSafe are active, the flow is:

```
GNM model
    └──► /gnm/cmd_vel_raw    (what GNM wants — unfiltered)
                │
         FleetSafe CBF-QP
         (reads /scan for obstacles, /odom for current speed)
                │
         /fleetsafe/cmd_vel_safe   (approved or corrected command)
                │
         /cmd_vel                  (robot actually moves)
```

Nothing writes directly to `/cmd_vel` in production mode except FleetSafe.
This ensures the robot can never receive an unsafe command, regardless of what
GNM produces.
