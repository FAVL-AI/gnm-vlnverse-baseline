# v2.4.1 — Visible Yahboom Isaac Stage and ROS 2 Publisher Scaffold

## Purpose

v2.4.1 documents the transition from an empty/broken Yahboom USD reference to a
visible, geometry-correct placeholder stage in Isaac Sim. It also defines the
required ROS 2 publisher/subscriber scaffold that must be wired via OmniGraph
before the first valid rosbag2 episode can be recorded.

No rosbag recording is claimed in v2.4.1. No training data is claimed. This
milestone is an engineering validation checkpoint only.

---

## Why the Previous Yahboom USD Was Empty

The file `assets/robots/yahboom_m3_pro/yahboom_m3pro_reference.usda` contained
only an Xform reference:

```usda
def Xform "YahboomM3Pro" (
    prepend references = @/home/favl/robotics/.../yahboom_m3pro.usd@
)
```

This reference is a path to a binary `.usd` file using an **absolute filesystem
path**. Two problems:

1. **The path is machine-specific.** On any other machine, or after moving the
   repo, the reference silently resolves to nothing — Isaac Sim shows an empty
   stage with no geometry, no joints, and no sensors.

2. **The binary USD itself was the output of an early Isaac URDF import that
   lacked OmniGraph action graphs.** The geometry may be present in the binary
   file, but there are no ROS 2 publishers or subscribers wired inside it. So
   even if the reference resolved correctly, `ros2 topic list` would still show
   only `/parameter_events` and `/rosout`.

## The Visible Placeholder Stage

To unblock engineering validation, a visible Yahboom placeholder was created
manually in Isaac Sim and saved as:

```
assets/robots/yahboom_m3_pro/yahboom_m3pro_visible_placeholder.usda
```

The stage contains:

| Prim path | Shape | Meaning |
|---|---|---|
| `/World/YahboomM3Pro/base_link` | Cube (0.75 × 0.45 × 0.18 m) | Robot chassis body |
| `/World/YahboomM3Pro/top_deck` | Cube (0.55 × 0.35 × 0.06 m) | Top mounting plate |
| `/World/YahboomM3Pro/camera_link` | Cube (0.10 × 0.16 × 0.10 m) | Camera sensor mount position |
| `/World/YahboomM3Pro/lidar_link` | Cylinder (r=0.16, h=0.08 m) | LiDAR sensor mount position |
| `/World/YahboomM3Pro/wheel_1` | Cylinder (r=0.11, h=0.08 m) | Front-right mecanum wheel |

## What This Stage Is and Is Not

### What it IS

- A visible, geometry-correct placeholder robot that appears in the Isaac Sim
  viewport when the USDA is loaded.
- The correct spatial layout: body, top deck, camera at front, LiDAR on top,
  four wheels at the corners, ground plane below.
- A self-contained, portable USDA file with no absolute filesystem paths.
- A reproducible starting point for adding OmniGraph ROS 2 publisher nodes.

### What it is NOT

- **Not photo-realistic** (and not intended to be). The robot is made of
  primitive cubes and cylinders with no textures, materials, or mesh detail.
  It is a block model, not a rendered simulation.
- **Not physics-enabled.** The placeholder prims have no collision, mass, or
  articulation properties. The robot cannot move or fall under gravity in its
  current form.
- **Not connected to any ROS 2 topics.** The ROS 2 Bridge extension is enabled,
  but no OmniGraph action graphs are wired. Running `ros2 topic list` after
  loading this stage shows only `/parameter_events` and `/rosout`. The five
  canonical topics are not published.
- **Not a valid rosbag2 source.** A recording made from this stage would contain
  no valid topic data. The episode validator (`validate_yahboom_episode.py`)
  would reject it because all five canonical topics would have `message_count = 0`.

## Required ROS 2 Publisher/Subscriber Scaffold

Five OmniGraph nodes must be added to the Isaac Sim stage before any valid
recording can begin. These are added via Isaac Sim's Visual Scripting interface
(Window → Visual Scripting → Action Graph).

### Node 1 — Camera Image Publisher

| Field | Value |
|---|---|
| Node type | `ROS2PublishImage` |
| Connected to | `/World/YahboomM3Pro/camera_link` camera sensor prim |
| Topic name | `/camera/image_raw` |
| Message type | `sensor_msgs/Image` |
| Used by | GNM visual input — context frames and goal frame |
| Gate | `ros2 topic echo /camera/image_raw` must show image messages |

**Why this topic:** GNM takes a sequence of recent RGB frames (context images)
and one goal image as input. Without `/camera/image_raw`, GNM has no visual
input and cannot navigate. This is the single most critical topic.

### Node 2 — Odometry Publisher

| Field | Value |
|---|---|
| Node type | `ROS2PublishOdometry` |
| Connected to | `/World/YahboomM3Pro/base_link` articulation prim |
| Topic name | `/odom` |
| Message type | `nav_msgs/Odometry` |
| Used by | GNM waypoint label computation, FleetSafe velocity estimate |
| Gate | `ros2 topic echo /odom` must show non-zero pose messages |

**Why this topic:** The GNM dataset converter reads `/odom` at each timestep to
compute the robot's position and derive waypoint labels. Without odometry, the
converter cannot compute where the robot is or where it needs to go next.
FleetSafe also reads odometry to estimate the robot's current velocity.

### Node 3 — Transform Tree Publisher

| Field | Value |
|---|---|
| Node type | `ROS2PublishTransformTree` |
| Connected to | `/World/YahboomM3Pro` prim tree |
| Topic name | `/tf` |
| Message type | `tf2_msgs/TFMessage` |
| Used by | Dataset converter robot pose, FleetSafe frame lookup |
| Gate | `ros2 run tf2_tools view_frames` must show base_link → odom chain |

**Why this topic:** The TF tree tells every node where each part of the robot is
in the world right now. The dataset converter uses `base_link → odom` TF to
compute the exact robot pose at each camera frame. Without TF, pose computation
fails and episode conversion cannot proceed.

### Node 4 — LiDAR Scan Publisher

| Field | Value |
|---|---|
| Node type | `ROS2PublishLaserScan` |
| Connected to | `/World/YahboomM3Pro/lidar_link` lidar sensor prim |
| Topic name | `/scan` |
| Message type | `sensor_msgs/LaserScan` |
| Used by | FleetSafe obstacle detection and CBF-QP safety constraint |
| Gate | `ros2 topic echo /scan` must show non-zero range array |

**Why this topic:** FleetSafe uses `/scan` to know how close obstacles are in
every direction. The CBF-QP safety constraint requires distance measurements to
determine whether a proposed movement command is safe. Without `/scan`, FleetSafe
cannot enforce any safety guarantee.

### Node 5 — cmd_vel Subscriber

| Field | Value |
|---|---|
| Node type | `ROS2SubscribeTwist` |
| Connected to | Wheel joint velocity controllers |
| Topic name | `/cmd_vel` |
| Message type | `geometry_msgs/Twist` |
| Used by | Robot movement — FleetSafe safe command execution |
| Gate | Publishing to `/cmd_vel` must cause visible wheel rotation |

**Why this topic:** `/cmd_vel` is the final step in the FleetSafe pipeline. After
FleetSafe applies the CBF-QP safety filter to the GNM raw command, the safe
command goes to `/cmd_vel`. The Yahboom chassis controller subscribes to this
topic and converts the Twist message into individual wheel speeds using mecanum
inverse kinematics. Without a `/cmd_vel` subscriber, the robot cannot move in
simulation.

## How to Add the OmniGraph Nodes

1. Load `assets/robots/yahboom_m3_pro/yahboom_m3pro_visible_placeholder.usda`
   in Isaac Sim.
2. Confirm the ROS 2 Bridge extension is enabled (Window → Extensions → search
   "ROS2 Bridge" → enable if not already).
3. Press Play.
4. Open the Action Graph editor (Window → Visual Scripting → Action Graph).
5. Add a new Action Graph.
6. For each of the five nodes above:
   - Click the + button to add a node.
   - Search for the node type (e.g. `ROS2PublishImage`).
   - Connect the input prim to the relevant robot prim.
   - Set the `topic_name` field to the canonical topic name.
7. After all five nodes are added, run:
   ```bash
   python3 scripts/gnm/verify_yahboom_live_topics.py --strict
   ```
   All five topics must pass. This opens Gate A5.
8. Save the stage over the existing file:
   ```
   assets/robots/yahboom_m3_pro/yahboom_m3pro_visible_placeholder.usda
   ```
