## Glossary — Every Term Explained

### ROS 2 — Robot Operating System 2

**ROS** stands for **Robot Operating System**.
**ROS 2** stands for **Robot Operating System 2**.

Despite the name, ROS 2 is not a full operating system like Windows or Linux. It is
a communication framework — a shared set of rules and tools that lets different
programs on a robot send messages to each other.

Simple analogy: ROS 2 is like an intercom system in a school. One room (the camera
program) announces what it sees. Another room (the navigation program) listens and
decides what to do. They never speak directly — they use named message channels
called **topics**.

ROS 2 is the industry standard for robotics research and is used on thousands of
different robots worldwide. Nothing about ROS 2 itself is specific to this project.

### URDF — Unified Robot Description Format

**URDF** stands for **Unified Robot Description Format**.

A URDF is a text file written in XML (a markup language similar to HTML) that
describes the physical structure of a robot. It lists every part of the robot —
the body, wheels, camera mount, lidar mount — and says how those parts connect to
each other, what each part weighs, and where it is positioned.

Simple analogy: If the Yahboom M3 Pro were a piece of flat-pack furniture, the URDF
is the full assembly manual showing which part goes where, how heavy each piece is,
and how they all connect.

Our URDF: `assets/robots/yahboom_m3_pro/yahboom_m3pro.urdf`

Joint names defined in the URDF (fixed — every part of the pipeline references these
by name):
- `fl_wheel_joint` — front left wheel
- `fr_wheel_joint` — front right wheel
- `rl_wheel_joint` — rear left wheel
- `rr_wheel_joint` — rear right wheel

### Xacro — XML Macro

**Xacro** stands for **XML Macro**.
**XML** stands for **Extensible Markup Language**.

Xacro is a smarter, templated version of URDF. It lets the robot description use
variables and reusable components so that changing one number (like the wheel radius)
updates it everywhere automatically.

Simple analogy: URDF is the finished printed manual. Xacro is the editable source
document that generates the manual.

Our Xacro: `ros2_ws/src/fleet_safe_description/urdf/yahboom_m3pro.urdf.xacro`

The Xacro supports a `use_sim` flag: `true` for Gazebo simulation, `false` for the
physical robot. Isaac Sim uses the plain URDF rather than the Xacro directly.

### USD — Universal Scene Description

**USD** stands for **Universal Scene Description**.

USD is a 3D scene file format created by Pixar (the film studio) and later adopted by
NVIDIA for robotics simulation. Isaac Sim works natively with USD files. A USD file
describes a full 3D world — where the robot is, how the room looks, where the lights
are, how surfaces behave under physics.

The import path for the Yahboom robot is:

```
Yahboom URDF
        ↓
Isaac Sim URDF Importer extension
        ↓
Yahboom USD stage
        ↓
Isaac Sim loads and simulates the robot
```

Our USD assets:
- `assets/robots/yahboom_m3_pro/yahboom_m3pro_reference.usda` — reference stage
- `assets/robots/yahboom_m3_pro/yahboom_m3pro.usd` — imported USD

Full import instructions: `docs/YAHBOOM_URDF_TO_USD_IMPORT.md`

### RGB — Red Green Blue

**RGB** stands for **Red Green Blue**.

An RGB camera records colour images. Each pixel in the image has three values —
one for how red it is, one for how green, one for how blue. Combined, they produce
a full colour photograph.

The Yahboom M3 Pro has a front-facing RGB camera. GNM (the navigation model) uses
the RGB camera feed as its primary sense. Without it, GNM cannot navigate.

### LiDAR — Light Detection and Ranging

**LiDAR** stands for **Light Detection and Ranging**.

A LiDAR sensor spins a laser beam and measures how long it takes for the beam to
bounce back from surfaces. The result is a precise distance measurement in every
direction around the robot — a 360-degree ring of distance values showing exactly
where walls, furniture, and people are.

LiDAR gives distances, not colours. A camera gives colours, not distances. They are
complementary: GNM uses the camera to steer, FleetSafe uses the LiDAR to stay safe.

### TF — Transform

**TF** stands for **Transform**.

TF is a continuously updated description of where every part of the robot is relative
to every other part, and where the robot itself is in the world. It is published as a
tree structure — for example:

```
world
  └── base_footprint  (robot footprint on the ground)
        └── base_link  (robot body at wheel-axle height)
              ├── camera_link  (where the camera is mounted)
              └── lidar_link   (where the LiDAR is mounted)
```

Any program that needs to know "where is the camera in world coordinates right now?"
queries the TF tree to get the answer.

TF matters for GNM training because the dataset converter uses TF to compute the
robot's exact pose at each frame. Without accurate TF, the waypoint labels used in
training are computed from the wrong position, and the trained model will be wrong.

### cmd_vel — Command Velocity

**cmd_vel** is short for **command velocity**.

`/cmd_vel` is the ROS 2 topic that drives the robot's wheels. A message on this
topic uses the message type `geometry_msgs/msg/Twist`. The key fields are:

```
linear.x   = forward/backward speed (metres per second)
linear.y   = sideways speed — only used by mecanum robots (metres per second)
angular.z  = turning speed (radians per second)
```

`/cmd_vel` is the final step in the safety pipeline. FleetSafe sits directly before
this topic. Nothing should publish to `/cmd_vel` in the production pipeline except
FleetSafe. If an unsafe command reaches `/cmd_vel`, the robot may move in a
dangerous direction.

### rosbag2

**rosbag2** is the standard ROS 2 recording tool.

It records every message published on every topic, with precise timestamps, to a
file on disk. Think of it as a black-box flight recorder for the robot. You can
replay the recording later and every topic re-publishes exactly as it was during
the original run.

For GNM training, a rosbag2 episode records:
- camera images from `/camera/image_raw`
- robot position from `/odom`
- sensor transforms from `/tf`
- obstacle distances from `/scan`
- movement commands from `/cmd_vel`

The recording is only valid if all five canonical topics have non-zero message counts.

### Mecanum Wheels

The Yahboom M3 Pro has four mecanum wheels. Mecanum wheels have small rollers
mounted at 45-degree angles around the rim, which allows the robot to move
forward, backward, sideways, or diagonally without rotating first. This is called
**holonomic motion**.

Standard wheeled robots (like TurtleBot) can only go forward and turn. Mecanum robots
can also slide sideways. This gives the Yahboom more flexibility in tight spaces.

The mecanum inverse kinematics converts a velocity command `[vx, vy, wz]` into
individual wheel speeds:

```
fl = (vx − vy − (lx + ly) × wz) / r
fr = (vx + vy + (lx + ly) × wz) / r
rl = (vx + vy − (lx + ly) × wz) / r
rr = (vx − vy + (lx + ly) × wz) / r
```

Where `lx` = half wheelbase (0.0775 m), `ly` = half track width (0.0850 m),
`r` = wheel radius (0.048 m).

Training on a mecanum robot in simulation and deploying on the same mecanum robot
in reality reduces the sim-to-real gap significantly compared to using a different
robot model.

## Why Yahboom Is Now the Main Robot

v2.1 proved the ROS 2 bridge works using Nova Carter, an NVIDIA research robot asset.
Nova Carter served as a **smoke test** — a quick pass/fail check that asks:

> "Can Isaac Sim launch, enable the ROS 2 bridge, and publish live topics at all?"

The answer is yes. Nova Carter's job is done.

From v2.3 onward, the main pipeline robot is the Yahboom ROSMASTER M3 Pro. This
matters because:

1. **Sim-to-real transfer:** training data should come from the same robot shape,
   kinematics, and sensor positions as the real deployment robot. Nova Carter has
   completely different geometry, sensors, and joint names.

2. **Topic names:** Nova Carter publishes non-standard topic names
   (e.g. `/front_stereo_camera/left/image_raw`, `/chassis/odom`). The GNM pipeline
   expects canonical names (`/camera/image_raw`, `/odom`). A rosbag recorded from
   Nova Carter cannot be converted into valid GNM training data without rewriting
   the entire converter.

3. **Correct kinematics:** the Yahboom has four mecanum wheels. Nova Carter has a
   completely different drive system. A model trained on Nova Carter data will not
   transfer to Yahboom.

4. **Real hardware parity:** Yahboom is the physical robot that will be deployed.
   Every step of the pipeline — sensor placement, command format, safety margins —
   must match the physical robot from the start.

## The Five Canonical Topics

These five topics must all be live before any rosbag2 recording is valid.

| Topic | Type | Purpose | Standard? |
|---|---|---|---|
| `/camera/image_raw` | `sensor_msgs/Image` | RGB camera images — primary GNM input | Yes, universal |
| `/odom` | `nav_msgs/Odometry` | Wheel encoder position and velocity estimate | Yes, universal |
| `/tf` | `tf2_msgs/TFMessage` | Robot part positions in world coordinates | Yes, universal |
| `/scan` | `sensor_msgs/LaserScan` | LiDAR obstacle distances | Yes, universal |
| `/cmd_vel` | `geometry_msgs/Twist` | Movement command — FleetSafe output | Yes, universal |

All five names are universal ROS 2 standards, not specific to this project.

### Why Strict Verification Must Pass Before Recording

The recording gate rule is:

```
All five canonical topics must publish non-zero messages
BEFORE any rosbag2 recording begins.
```

If any topic is missing and recording starts anyway, the resulting rosbag will be
incomplete. The GNM dataset converter will either fail or produce wrong training
labels. No amount of post-processing can recover missing sensor data.

The verification gate prevents wasted recording sessions and corrupted training data.

### Camera Alias — Remapping Required for Real Robot

The physical Yahboom M3 Pro hardware driver publishes the camera on:
```
/camera/color/image_raw
```

The GNM pipeline and Isaac Sim use:
```
/camera/image_raw
```

Before connecting the real robot, a topic remap must be added to the launch file:
```python
remappings=[("/camera/color/image_raw", "/camera/image_raw")]
```

This remap is already documented in `scripts/gnm/check_yahboom_topic_contract.py`.

## Isaac Sim Import Sequence

Before the five topics can be verified live, the Yahboom USD stage must be loaded
in Isaac Sim. The ordered steps are:

1. Launch Isaac Sim (`~/.local/share/ov/pkg/isaac-sim-*/isaac-sim.sh`)
2. Enable the URDF Importer extension (Window → Extensions → search "URDF")
3. Import the Yahboom URDF (File → Import → URDF):
   `assets/robots/yahboom_m3_pro/yahboom_m3pro.urdf`
4. Save as USD: `assets/robots/yahboom_m3_pro/yahboom_m3pro.usd`
5. Enable the ROS 2 Bridge extension (Window → Extensions → search "ROS2 Bridge")
6. Add OmniGraph action graphs:
   - Camera publisher → `/camera/image_raw`
   - Odometry publisher → `/odom`
   - TF tree publisher → `/tf`
   - LiDAR scan publisher → `/scan`
   - Twist subscriber → `/cmd_vel`
7. Press Play
8. From a terminal, run strict topic verification:
   ```bash
   python3 scripts/gnm/verify_yahboom_live_topics.py --strict
   ```

Full import reference: `docs/YAHBOOM_URDF_TO_USD_IMPORT.md`

simulation first before language grounding can be layered on top.

Track B completion gates are defined separately in
`docs/track_a_track_b_completion_gates.md`. Track B must not be claimed complete
unless its own evidence, tests, and claim boundaries exist independently of Track A.
