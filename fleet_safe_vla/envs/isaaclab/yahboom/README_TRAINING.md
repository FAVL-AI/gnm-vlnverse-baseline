# Yahboom M3Pro — Isaac Lab RL Training Roadmap

**Status: SCAFFOLD — do not train until Stage 0 passes.**

This document is the authoritative training roadmap for the Yahboom
RosMaster M3Pro in Isaac Lab. It covers the 6 training stages, the
observation/action space, reward design, asset prerequisites, and the
path from simulation to real hardware.

---

## Quick-start checklist

Run this before anything else:

```bash
./scripts/isaaclab/train_yahboom.sh --stage 0
```

This checks all required assets without starting Isaac Sim.

If Stage 0 passes, proceed to Stage 1:

```bash
./scripts/isaaclab/train_yahboom.sh --stage 1
```

## Action space

**3-DoF normalised velocity command, scaled to physical limits.**

| Dim | Symbol | Range (norm) | Physical range |
|---|---|---|---|
| 0 | `vx_norm` | [−1, 1] | [−0.5, 0.5] m/s (forward/backward) |
| 1 | `vy_norm` | [−1, 1] | [−0.5, 0.5] m/s (strafe left/right) |
| 2 | `wz_norm` | [−1, 1] | [−1.0, 1.0] rad/s (yaw) |

The `M3ProVelocityActionCfg` (to be implemented in `m3pro_env.py`) converts
these to 4-wheel velocity targets via inverse mecanum kinematics:

```
fl = (vx - vy - (lx + ly) * wz) / r
fr = (vx + vy + (lx + ly) * wz) / r
rl = (vx + vy - (lx + ly) * wz) / r
rr = (vx - vy + (lx + ly) * wz) / r
```

where `lx = 0.0775 m`, `ly = 0.0850 m`, `r = 0.048 m`.

## Reward function

### Stage 1 baseline (all-step, additive)

| Term | Weight | Formula | Purpose |
|---|---|---|---|
| `track_vx` | +2.0 | exp(−err²/0.0625) | Track commanded linear x velocity |
| `track_vy` | +1.5 | exp(−err²/0.0625) | Track commanded strafe velocity |
| `track_wz` | +1.0 | exp(−err²/0.0625) | Track commanded yaw rate |
| `lin_vel_z` | −0.5 | ‖vz‖² | Penalise vertical bounce |
| `ang_vel_xy` | −0.05 | ‖ωxy‖² | Penalise pitch/roll oscillation |
| `flat_orient` | −1.0 | ‖g_projected‖² | Penalise tipping |
| `action_rate` | −0.01 | ‖aₜ − aₜ₋₁‖² | Penalise jerky commands |

**Convergence target:** mean episodic reward > 3.5 per step (≈ tracking reward
dominates).

### Stage 2 additions

| Term | Weight | Formula | Purpose |
|---|---|---|---|
| `goal_reached` | +10.0 | sparse, dist < 0.2 m | Reach waypoint |
| `heading_align` | +0.5 | cos(θ_error) | Face the goal |
| `goal_progress` | +1.0 | −Δdist / dt | Reward getting closer |

### Stage 3 additions

| Term | Weight | Formula | Purpose |
|---|---|---|---|
| `obstacle_prox` | −2.0 | max(0, d_safe − d_min) | Penalise entering 0.3 m zone |
| `collision` | −10.0 | sparse, contact detected | Episode-ending penalty |

### Stage 4 additions

| Term | Weight | Formula | Purpose |
|---|---|---|---|
| `cbf_violation` | −5.0 | max(0, −h) where h = CBF barrier | Safety constraint |
| `estop_penalty` | −20.0 | sparse, estop triggered | Hard safety violation |

## Training stages

### Stage 0 — Asset validation

```bash
./scripts/isaaclab/train_yahboom.sh --stage 0
```

- No Isaac Sim launched; runs in <5 seconds
- Checks URDF, MJCF, obs adapter, USD cache
- Prints asset checklist with actionable error messages
- **Gate:** all required assets must exist before Stage 1

---

### Stage 1 — Random cmd_vel tracking

**Goal:** policy executes arbitrary [vx, vy, wz] commands accurately.

```bash
./scripts/isaaclab/train_yahboom.sh --stage 1
# With fewer envs for debugging:
./scripts/isaaclab/train_yahboom.sh --stage 1 --num-envs 128 --headless false
```

| Parameter | Value |
|---|---|
| Envs | 1024 |
| Episode length | 20 s |
| Control rate | 10 Hz |
| Sim rate | 100 Hz |
| Decimation | 10 |
| PPO iterations | 2000 |
| Rollout steps/env | 24 |
| Expected wall-clock | ~2–3 h (RTX 4080 SUPER) |

Commands are sampled uniformly from the full velocity range and resampled
every 4–8 seconds. The robot starts in a random pose each episode.

**Convergence signal:** TensorBoard `rew/track_vx` + `rew/track_vy` both
plateau above 0.85 (normalised). Run:

```bash
tensorboard --logdir logs/isaaclab/yahboom_m3pro/stage1
```

---

### Stage 2 — Waypoint navigation

**Goal:** navigate to sequentially spawned waypoints on a flat 6×6 m floor.

```bash
./scripts/isaaclab/train_yahboom.sh --stage 2 --resume
```

Warm-starts from the best Stage 1 checkpoint (`--resume`). Adds `goal_vec`
to the observation and `goal_reached` + `heading_align` rewards.

**New files needed before Stage 2:**
- Waypoint spawning logic in `m3pro_env.py`
- `goal_vec` observation term

---

### Stage 3 — Obstacle avoidance

**Goal:** navigate to waypoints while avoiding static obstacles.

```bash
./scripts/isaaclab/train_yahboom.sh --stage 3 --resume
```

Adds 360-dim lidar scan to observations. Scene populated with 5–20 randomly
placed boxes/cylinders. Network widened to [512, 256, 128].

**New files needed before Stage 3:**
- Obstacle spawning in scene config
- `lidar_scan` observation term (raycast from robot frame)
- Larger variant of `M3ProSceneCfg`

---

### Stage 4 — Fleet-Safe CBF safety layer

**Goal:** policy respects the CBF safety constraints from `fleet_safety/`.

```bash
./scripts/isaaclab/train_yahboom.sh --stage 4 --resume
```

Fine-tunes Stage 3 policy with CBF reward terms. Safety state (barrier value,
min obstacle distance, estop) added to observations. The CBF filter from
`fleet_safe_vla/fleet_safety/yahboom_cbf.py` runs in the reward computation.

This stage bridges simulation and real hardware: the same CBF parameters are
used in both the Isaac Lab environment and the ROS2 safety node.

## Hardware transfer

**Sim → Real transfer path (after Stage 4):**

1. Export policy as TorchScript:
   ```python
   torch.jit.save(policy.actor, "m3pro_policy.pt")
   ```

2. Deploy to real M3Pro via the ROS2 bridge:
   ```
   m3pro_policy.pt → obs_adapter_m3pro.py → /cmd_vel_nominal
                                           → safety_node → /cmd_vel → robot
   ```

3. Verify with FleetSafe dashboard (Real M3Pro tab):
   ```bash
   ./scripts/real_robot/run_m3pro_ros2_bridge.sh
   # http://localhost:8080/yahboom → Real M3Pro
   ```

4. Test at 0.15 m/s (conservative) before increasing to 0.5 m/s.

**Key sim-to-real gap mitigations:**
- Domain randomisation in `EventCfg` (mass ±20%, friction ±30%, pushes)
- Gaussian noise on all observations matches real sensor noise
- Mecanum wheel slip is idealised in simulation — expect some degradation
  in real strafe accuracy until friction params are calibrated
