# Bo Follow-up — Start/Goal Evidence, Success Rate, and Custom Scene

---

## Q: When I click START or GOAL in the Stage tree, why do I not see images?

The original START and GOAL prims in `/World/GNM_Replay` were geometric marker
prims only (USD Sphere primitives).
Clicking them in the Stage panel showed only the standard USD transform properties
(position, orientation, scale) — not the actual camera images or episode statistics.

I have now addressed this in two ways:

1. **Custom USD attributes on START and GOAL prims**  
   The updated `scripts/gnm/replay_gnm_demo.py` writes the following custom
   attributes onto the START and GOAL prims before replay starts:

   - `gnm:scene_id` — e.g. `kujiale_0118`
   - `gnm:episode_id` — e.g. `kujiale_0118_25_3`
   - `gnm:start_position` — `[x, y]` in metres
   - `gnm:goal_position` — `[x, y]` in metres
   - `gnm:start_yaw_rad` — heading angle at start
   - `gnm:goal_yaw_rad` — heading angle at goal frame
   - `gnm:path_length_m` — total expert trajectory length
   - `gnm:n_frames` — number of recorded frames
   - `gnm:initial_dist_m` — Euclidean distance from start to goal
   - `gnm:success_radius_m` — 3.0 m (evaluation success threshold)
   - `gnm:start_image_path` — path to `0.jpg`
   - `gnm:goal_image_path` — path to the final frame JPG

   When Bo clicks START or GOAL, the USD Property panel now shows all of these
   fields alongside the standard transform.

2. **Image panels in the viewport**  
   The updated demo creates two textured plane prims under `/World/GNM_Replay`:
   - `CURRENT_OBS_PANEL` — shows the current observation image
   - `GOAL_IMAGE_PANEL` — shows the goal image

   These are PNG textures generated from the actual trajectory frames using
   Python Imaging Library, written to `results/figures/` before the scene opens.

## Q: What are the start and goal statistics for each trajectory?

For each replayed trajectory, the following statistics are computed from
`traj_data.pkl` and `episode_info.json`:

| Field | Description |
|-------|-------------|
| `start_x`, `start_y` | Floor-plane start position in metres |
| `goal_x`, `goal_y` | Floor-plane goal position in metres |
| `start_yaw_rad` | Robot heading at frame 0 (radians) |
| `goal_yaw_rad` | Robot heading at goal frame (radians) |
| `path_length_m` | Total expert path length along the trajectory |
| `n_frames` | Number of recorded RGB camera frames |
| `initial_dist_m` | Straight-line distance from start to goal |
| `success_radius_m` | 3.0 m — the radius within which the robot must stop |

These are available:
- As custom USD attributes on START/GOAL prims (visible in Property panel)
- Printed in the terminal when the replay script runs
- Computed by `scripts/gnm/inspect_trajectory_labels.py`

## Q: Why is Success Rate 20 percent?

Success Rate (SR) = successful episodes ÷ total evaluated episodes.

For the current baseline:
- **Evaluated: 15 validation episodes**
- **Successful: 3 episodes** (final position within 3.0 m of goal)
- **SR = 3 / 15 = 20.0%**

The General Navigation Model stops navigating when its predicted distance-to-goal
(`dist_pred`) drops below the stop threshold (0.15).
In episodes where SR ≠ OSR (Oracle SR = 46.7% = 7/15), the robot was close to
the goal at some point but the distance head predicted the goal was still far,
so it kept walking and overshot.

A full per-episode breakdown is in:
`results/eval_episode_breakdown_tracka.md`

Generated by:
```bash
python3 scripts/gnm/explain_eval_success_rate.py \
    --checkpoint checkpoints/gnm_base/best.pt \
    --output results/eval_episode_breakdown_tracka.md
```

## Q: Can you create your own scene separate from Vision-Language Navigation Verse?

Yes.
`scripts/gnm/create_custom_gnm_scene.py` builds a small custom office scene in
NVIDIA Isaac Sim from primitive USD shapes — no VLNVerse scene assets are used.

The scene contains:
- Flat floor plane (grey, 10 m × 10 m)
- Four walls (white cubes)
- One obstacle (table-like cube)
- START marker (green sphere)
- GOAL marker (red sphere)
- Planned straight-line + turn path overlaid as cyan spheres
- RGB camera at the robot's eye height

The script:
1. Opens Isaac Sim with a blank stage
2. Creates the geometry from USD primitives
3. Places the camera at a start position
4. Moves the camera along the planned path, capturing one RGB frame per step
5. Saves `traj_data.pkl` with `position` and `yaw` arrays
6. Saves RGB frames as numbered JPGs
7. Saves `episode_info.json`

Outputs:
```
datasets/custom_gnm_scene/train/custom_office_0001/traj_data.pkl
datasets/custom_gnm_scene/train/custom_office_0001/0.jpg … N.jpg
datasets/custom_gnm_scene/train/custom_office_0001/episode_info.json
results/figures/custom_scene_start.png
results/figures/custom_scene_goal.png
```

Run:
```bash
conda run -n isaac python scripts/gnm/create_custom_gnm_scene.py
```

This demonstrates that the General Navigation Model data pipeline is not locked
to VLNVerse assets — any scene that can be loaded in Isaac Sim can be used.

## Q: How does this connect to the GNM architecture figure?

The General Navigation Model architecture uses four inputs:
1. Current observation image (what the robot sees now)
2. Goal image (what the robot should see at the goal)
3. Context images (previous N frames for temporal awareness)
4. Shared feature encoder → distance head + action head

The Isaac Sim replay demo maps each of these to visible evidence:

| Architecture component | Demo evidence |
|------------------------|--------------|
| Current observation image | `CURRENT_OBS_PANEL` textured plane in viewport |
| Goal image | `GOAL_IMAGE_PANEL` textured plane in viewport |
| Start position | `START` green sphere + custom USD attributes |
| Goal position | `GOAL` red sphere + custom USD attributes |
| Expert trajectory | Cyan `path_XXXX` sphere chain |
| Robot motion | `ROBOT_MARKER` blue sphere animated along path |
| Architecture overview | `results/figures/gnm_architecture_evidence_panel.png` |

When Bo opens the Isaac Sim replay scene, every component of the architecture
figure has a corresponding visible artifact in the viewport.
