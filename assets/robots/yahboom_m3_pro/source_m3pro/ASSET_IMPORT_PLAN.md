## Physical measurement protocol (before Stage 1)

### Tools needed
- Digital kitchen scale (±1 g resolution)
- Digital calipers (±0.1 mm)
- Steel tape measure

### Procedure

```
1. MASSES
   a. Place complete robot on scale.  Record total_mass.
   b. Remove each of the 4 wheels (usually 1–2 screws).
      Weigh each wheel.  Record wheel_mass_fl, ..._rr.
      Expect: ~0.130–0.180 kg per wheel.
   c. body_mass = total_mass − sum(wheel_masses)

2. WHEEL RADIUS
   Using calipers, measure wheel outer diameter at the tread.
   radius = diameter / 2.
   Expect: ~0.048 m.  Update URDF if > ±2 mm off.

3. WHEELBASE (front↔rear axle distance)
   Lay a ruler along the robot side.
   Measure centre-to-centre of front and rear axle bolts.
   Expect: ~0.155 m.

4. TRACK WIDTH (left↔right wheel centre)
   Measure from wheel centre to wheel centre across the robot.
   Expect: ~0.170 m.

5. SENSOR HEIGHTS (from floor with robot on flat surface)
   a. Lidar scan plane height (bottom of lidar housing):  expect ~0.160 m
   b. Camera lens centre height:                           expect ~0.130 m
   c. Camera forward offset from robot centre:             expect ~0.100 m

6. BODY INERTIA (if CAD not available)
   Bifilar pendulum method:
   https://en.wikipedia.org/wiki/Bifilar_suspension
   Or use CAD export from Yahboom STEP file (see §Official assets below).
```

### Updating the URDF after measurement

Edit `urdf/yahboom_m3pro.urdf`:
1. Update `<mass value="..."/>` in `base_link` and `fl_wheel` (and copy to fr/rl/rr).
2. Update joint origins `xyz="lx ly 0"` if wheelbase/track differ.
3. Recompute inertias with box/cylinder formulas (comments in URDF show formulas).
4. Re-run validator: `python scripts/yahboom/validate_m3pro_assets.py`

## Official asset sources

### Option A — Yahboom official ROS2 package (preferred)

Yahboom publishes ROS2 driver packages for their robots.  Search on GitHub for:
- Organization: `YahboomTechnology` or `yahboom`
- Repository names containing: `ROSMASTER`, `m3pro`, or `rosmaster_x3` (X3 package
  may include M3Pro description as a variant)

If found, their package typically provides:
```
yahboom_description/
└── urdf/
    └── rosmaster_m3pro.urdf.xacro   ← copy joint/link values from here
```

To adapt their XACRO to our URDF:
```bash
# Expand xacro (if ROS2 Humble is available):
ros2 run xacro xacro path/to/rosmaster_m3pro.urdf.xacro > /tmp/m3pro_expanded.urdf

# Then diff against ours to find differing joint positions:
diff /tmp/m3pro_expanded.urdf \
     fleet_safe_vla/robots/yahboom/m3pro/urdf/yahboom_m3pro.urdf

# Copy measured joint origins and inertials into our URDF.
```

Important: if their URDF uses different joint names (e.g. `wheel_fl` instead of
`fl_wheel_joint`), keep our names — the obs_adapter, env_cfg, and ROS2 bridge
all reference the joint names in `robot_contract_m3pro.yaml`.

### Option B — Yahboom product page

The M3Pro product page (search "Yahboom ROSMASTER M3Pro") often provides:
- STEP/STP file for the full assembly
- PDF mechanical drawing with measured dimensions

Convert STEP to URDF:
```bash
# Extract STL meshes from STEP (FreeCAD or OpenSCAD):
freecad-console convert_step_to_urdf.py rosmaster_m3pro.step

# Or: use solidworks_urdf_exporter if you have SolidWorks access.
```

### Option C — Derive from X3 package (quick, lower fidelity)

The X3 and M3Pro share the same wheel radius and similar body proportions.
The main differences are:
- X3: 2 drive wheels + 2 casters → M3Pro: 4 mecanum wheels
- X3: differential drive → M3Pro: holonomic

The current URDF already does this derivation. It is sufficient for Stage 0–2.

## MJCF creation plan (for MuJoCo validation)

After the URDF is finalised, generate the MJCF:

```bash
python3 -c "
import mujoco
m = mujoco.MjModel.from_xml_path(
    'fleet_safe_vla/robots/yahboom/m3pro/urdf/yahboom_m3pro.urdf')
print(f'nq={m.nq}  nu={m.nu}  ngeom={m.ngeom}')
"
```

Then hand-edit `mjcf/yahboom_m3pro.xml` to:
1. Add mecanum friction approximation (see `mjcf/PLACEHOLDER.md`)
2. Add velocity actuators (kv matching X3 baseline)
3. Add sensors (accelerometer, gyro, jointpos, jointvel)

## ros2_control hardware plugin

The URDF declares `yahboom_hardware/YahboomM3ProHardware`.
This plugin must be created before the robot can be driven with ros2_control.

Options:
1. **Use mock_components** for simulation-only use:
   Replace the `<plugin>` line in the URDF with:
   ```xml
   <plugin>mock_components/GenericSystem</plugin>
   ```
2. **Implement real plugin** in `ros2_ws/src/yahboom_hardware/` for physical robot.
   Reference: `ros2_controllers/diff_drive_controller` and the existing X3 hardware
   plugin pattern in the Yahboom ROS2 package (if obtained from Option A above).

## Remaining blockers

| Blocker | Impact | Resolution |
|---|---|---|
| Inertials are box approximations | Stage 1 physics accuracy ±30% | Measure masses; recompute |
| Camera x-offset estimated | Stage 3 camera obs slightly off | Measure with ruler |
| No mesh geometry | Visual only — no mesh fidelity in GUI | Obtain Yahboom STEP file |
| MJCF placeholder inertials | Stage 1 physics accuracy limited | Replace with measured values (see Stage 1) |
| ros2_control plugin stub | Physical robot ROS2 bringup | Implement or use mock_components |
